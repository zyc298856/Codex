## 三、板端推理实测

### 测试环境

- 测试视频：`/home/ubuntu/public_videos/anti_uav_fig1.mp4`（130 帧）
- 置信度阈值：`score_thresh=0.35`
- NMS 阈值：`nms_thresh=0.45`（默认）
- 性能分析：`RK_YOLO_PROFILE=1`
- 每次运行前确认模型加载成功（输出 tensor shape [1, 5, 8400]）

### 5 组对比测试结果

| 编号 | 模型 | 预处理方式 | 帧有检测 | 总检测数 | 平均推理时间 |
|------|------|-----------|---------|---------|------------|
| **A** | FP baseline | OpenCV | 130/130 | 9,352 | 86.48 ms |
| **B** | Hybrid INT8 head230 | OpenCV | 130/130 | 114,359 | 187.71 ms |
| **C** | FP baseline | RGA letterbox | 130/130 | 9,352 | 75.57 ms |
| **D** | Hybrid INT8 head230 | RGA | 130/130 | 114,359 | 195.85 ms |
| **E** | Full INT8 | OpenCV | **0/130** | **0** | 31.68 ms |

### 关键发现

#### 1. Full INT8 = 0 检测（实测确认）

```
Test E: Full INT8 model
  model input: [1, 640, 640, 3] float16
  model output: [1, 5, 8400] float16
  frames_with_detections: 0 / 130
  total_detections: 0
  avg_inference: 31.68 ms
```

**结论**：Full INT8 量化导致检测头精度完全丧失，130 帧无一检测。速度虽快（31.68ms），但毫无实用价值。这是板端实测的确定性结论。

#### 2. Hybrid INT8 检测数异常高

Hybrid INT8 总检测数 114,359 远超 FP 的 9,352。原因分析：
- Hybrid 模型的检测头保留 FP 精度，但 backbone 的 INT8 量化改变了特征分布
- 输出的置信度分布发生偏移，大量低置信度候选框超过 0.35 阈值
- 平均每帧约 879 个检测（114359/130），说明后处理可能产生大量重复框
- FP 基线平均每帧约 72 个检测（9352/130），更为合理

**影响**：Hybrid INT8 虽然能产生检测，但检测质量与 FP 有显著差异。在论文中应诚实说明这一现象。

#### 3. RGA 预处理验证通过

Test C（FP + RGA letterbox）的检测结果（9,352）与 Test A（FP + OpenCV）**完全一致**，证明：

- RGA letterbox 预处理路径功能正确
- RGA 预处理不改变检测结果
- RGA 使 FP 推理从 86.48ms 降至 75.57ms（**12.6% 加速**）

板端运行确认的 RGA 相关输出：

```
RGA letterbox used: OpenCV cvtColor/resize/copyMakeBorder skipped
```

这证明 RGA letterbox **完全绕过**了 OpenCV 的 cvtColor、resize、copyMakeBorder 三个操作，由硬件完成颜色转换、缩放和 padding。

#### 4. GPIO 状态确认

所有 5 组测试均输出：

```
gpio_alarm_path=<disabled>
```

**结论**：GPIO 功能在当前构建中明确禁用，与代码审查结论一致——GPIO 硬件驱动接口未实现。

## 七、答辩建议（基于板端实测）

### 必须准备的关键问答

**Q1：INT8 量化效果怎么样？**

> "Full INT8 量化后检测数降为 0，分析原因是 YOLOv10 检测头对量化敏感。Hybrid INT8（检测头保留 FP）恢复了检测能力，但实测推理速度从 FP 的 86ms 增加到 188ms，并未实现加速。最终演示采用 FP RKNN + RGA 硬件预处理方案，达到 75ms/帧（13.2 FPS），比纯 OpenCV 预处理提升 12.6%。"

**Q2：GPIO 做了吗？**

> "系统实现了完整的软件告警逻辑——画面叠加告警标识、告警事件日志记录。GPIO 硬件驱动接口设计了兼容层（gpio_alarm_path 参数），但因缺少实体外设（LED/蜂鸣器/继电器），未进行硬件验证，作为后续工作。"

**Q3：RGA 的作用是什么？**

> "RGA 是 RK3588 的硬件图像加速器。实测证明 RGA letterbox 路径完全绕过 OpenCV 的颜色转换、缩放和 padding 操作，且检测结果与 OpenCV 完全一致（9352 vs 9352），推理速度提升 12.6%。系统通过 RK_YOLO_REQUIRE_RGA=1 环境变量提供严格模式，确保 RGA 不可用时直接报错而非静默回退。"

**Q4：为什么 Hybrid INT8 比 FP 还慢？**

> "Hybrid INT8 的 backbone 使用 INT8 计算，但检测头保留 FP16。这导致两个问题：一是混合精度计算本身有额外开销；二是 INT8 backbone 的特征分布偏移导致检测头输出了大量低置信度候选框（114359 vs FP 的 9352），后处理（NMS）耗时大幅增加。这说明量化不仅仅是算术精度的问题，还会改变模型的输出分布。"

- GPIO → "设计了接口规范，实体外设验证为后续工作"
- INT8 → "Full INT8 精度丧失，Hybrid INT8 未实现加速，最终采用 FP + RGA 方案"
